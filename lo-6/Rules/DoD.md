# Rule - Definition of Done (DoD)

## 功能完成的标准清单

每个功能在标记为"完成"之前，必须满足以下所有条件：

## 代码质量
- [ ] 所有单元测试通过
- [ ] 所有集成测试通过
- [ ] 测试覆盖率达到要求（关键路径 100%，整体 ≥ 80%）
- [ ] 无 ESLint/TSLint 错误
- [ ] 无 TypeScript 类型错误
- [ ] 代码已通过 Code Review（如适用）

## 功能完整性
- [ ] 所有验收标准（AC）已实现
- [ ] 前后端接口已联调通过
- [ ] 错误处理已实现
- [ ] 边界情况已处理

## 文档同步
- [ ] 需求规格文档已更新（如有变更）
- [ ] 设计规格文档已更新（如有变更）
- [ ] UI 规格文档已更新（如有变更）
- [ ] API 文档已更新（如适用）
- [ ] README 已更新（如适用）

## 版本控制
- [ ] 代码已提交到 Git
- [ ] Commit message 清晰描述变更
- [ ] 无未追踪的临时文件
- [ ] 分支已推送到远程仓库

## 用户验收
- [ ] 功能已部署到测试环境（如适用）
- [ ] 用户已手工测试并确认
- [ ] 无阻塞性 bug

## 非功能性要求
- [ ] 性能符合预期（无明显卡顿）
- [ ] 无安全漏洞（SQL注入、XSS等）
- [ ] 无内存泄漏
- [ ] 日志记录完整

## 如何使用此规则

**在开发过程中：**
- 将此清单作为开发指南，确保不遗漏任何项目

**在功能完成时：**
- 逐项检查，全部通过后才能标记为完成
- 如有未通过项，继续开发直到满足

**在验收时：**
- 用户和开发者共同确认清单
- 记录验收时间和验收人
