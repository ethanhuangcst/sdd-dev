# Rule - Test Strategy

## 测试策略和质量保证标准

## 测试金字塔原则

遵循测试金字塔结构，从下到上：
- **单元测试（70%）** - 快速、隔离、大量
- **集成测试（20%）** - 验证模块间交互
- **端到端测试（10%）** - 验证完整用户流程

## 单元测试要求

### 覆盖范围
- 所有业务逻辑函数必须有单元测试
- 所有工具函数必须有单元测试
- 关键路径覆盖率 100%
- 整体覆盖率 ≥ 80%

### 测试原则
- 每个测试只验证一个行为
- 测试命名清晰描述测试场景：`should_[expected]_when_[condition]`
- 使用 AAA 模式：Arrange（准备）、Act（执行）、Assert（断言）
- 测试必须独立，不依赖执行顺序
- 测试必须可重复，每次运行结果一致

### 不使用 Mock 的场景
- 数据库连接：使用真实数据库或测试数据库
- API 调用：使用真实 API 或测试环境 API
- AI 服务：使用真实 AI 连接或测试 API key

### 何时使用 Mock
- 外部第三方服务（支付、短信等）
- 时间相关函数（Date.now()）
- 随机数生成
- 文件系统操作（仅在必要时）

## 集成测试要求

### 覆盖范围
- API 端点的请求/响应
- 数据库操作的 CRUD
- 前后端接口联调
- 认证和授权流程

### 测试原则
- 使用真实的数据库连接
- 测试前清理数据，测试后恢复
- 验证完整的数据流转
- 包含错误处理场景

## 端到端测试要求

### 覆盖范围
- 关键用户流程（登录、核心功能）
- 跨页面的交互流程
- 表单提交和验证

### 测试原则
- 模拟真实用户操作
- 在真实浏览器中运行（如使用 Playwright）
- 包含正常流程和异常流程

## 测试数据管理

### 原则
- 使用独立的测试数据库
- 每个测试使用独立的测试数据
- 测试后清理数据，避免污染

### 实现
- 使用 fixtures 或 factories 生成测试数据
- 使用事务回滚或数据库重置
- 不使用生产数据

## TDD 工作流

### Red-Green-Refactor 循环
1. **Red（红灯）** - 先写测试，测试失败
2. **Green（绿灯）** - 写最小代码使测试通过
3. **Refactor（重构）** - 优化代码，保持测试通过

### 实践要点
- 先写测试，再写实现
- 每次只关注一个测试
- 测试通过后立即 commit
- 重构时保持测试通过

## 持续集成

### CI 流程
- 每次 push 自动运行所有测试
- 测试失败则阻止合并
- 生成测试覆盖率报告

## 何时可以跳过测试

**绝对不能跳过：**
- 业务逻辑代码
- API 端点
- 数据处理函数
- 认证授权逻辑

**可以考虑跳过：**
- 纯展示型组件（无逻辑）
- 配置文件
- 类型定义文件

## 测试失败处理

### 原则
- 测试失败时，优先修复代码而非修改测试
- 如果测试本身有问题，先修复测试
- 不要为了通过测试而降低测试标准
- 不要注释掉失败的测试

### 调试步骤
1. 阅读测试失败信息
2. 理解测试期望的行为
3. 检查实现代码是否符合预期
4. 使用调试工具定位问题
5. 修复后确保所有测试通过

## 如何应用此规则

**在编写代码前：**
- 先写测试，明确预期行为

**在开发过程中：**
- 遵循 TDD 循环
- 保持测试通过
- 及时 commit

**在代码审查时：**
- 检查测试覆盖率
- 验证测试质量
- 确保遵循测试原则
